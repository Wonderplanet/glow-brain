## セクション記述ガイドライン

### 概要セクション

- テーブルの「役割」を1文で端的に述べる
- 他テーブルとの関係（参照する側か参照される側か）を明記
- このテーブルが「何を管理して、何をしないか」の境界を明確にする
- 複雑な参照関係はコードブロックで視覚化する

### 全カラム一覧セクション

- DBスキーマの `data_type` を記載する（varchar, bigint, integer, boolean, enum など）
- 必須かどうかは `is_nullable` が `NO` なら `○`、`YES` なら空欄
- `ENABLE` カラムは必ず最初に記載（`e` = 有効の説明を付ける）
- enumカラムは「後述」と書いて専用セクションで詳しく解説する
- 関連テーブルへの外部キーは「参照先テーブル名.カラム名」を明記する

### enum/フラグセクション

- テーブルに enum型カラムがある場合、カラムごとにセクションを作る
- enum値の「ゲーム上での意味」を具体的に説明する
- クライアント実装（.csファイル）でenum定義が見つかった場合はその値と照合する
- 値が空（NULL/空文字）の場合の挙動も記述する

### 命名規則セクション

- `id` カラムに命名パターンがある場合のみ記述（ない場合は省略）
- 実データから命名パターンを帰納的に説明する
- 例外的なIDパターンがある場合も記述する

### 他テーブルとの連携セクション

- このテーブルを参照している他テーブル（逆参照）も記述する
- データフローが複雑な場合はコードブロックで図示する
- 外部キーカラムがない場合は「このテーブルは単独で完結する」と明記する

### 実データ例セクション

- 最低2パターン、最大5パターンを記述
- シンプルなケースと複雑なケースの両方を含める
- 実際のCSVデータから取得した値を使う（架空データは使わない）
- データの後に「このパターンの意味」を箇条書きで説明する

### 設定時のポイントセクション

- 5〜8項目のポイントを記述する
- 「〜してはいけない」「〜に注意」という形式で実践的に書く
- よくあるミスや、分かりにくい仕様を重点的に取り上げる
- コードブロックで具体的な値の対応を示す
