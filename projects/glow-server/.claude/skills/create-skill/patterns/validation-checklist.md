# スキル完成後の検証チェックリスト

スキル作成完了後に確認すべき項目をまとめたチェックリストです。

## SKILL.md の検証

### 基本構造

- [ ] **行数は30-50行（最大100行以下）か**
  - 現在の行数: ___行
  - 100行を超える場合は内容を参照ファイルに移動

- [ ] フロントマター（---で囲まれた部分）が正しいか
  - `name` フィールドが存在するか
  - `description` フィールドが存在するか

### nameフィールド

- [ ] **ジェランド形（動詞 + -ing）を使用しているか**
  - ✅ 良い例: `Creating Code Skills`, `Implementing API Endpoints`
  - ❌ 悪い例: `Code Skill Creation`, `API Endpoint Implementation`

### descriptionフィールド

- [ ] **使用タイミングと機能説明が1行で記載されているか**
  - 前半: 使用タイミング（「〜が必要な時に使用」「〜する際に使用」）
  - 後半: 機能説明（何を実装するか、何が得られるか）

- [ ] **最大1024文字以内か**
  - 現在の文字数: ___文字

- [ ] 具体的で分かりやすい内容か
  - ✅ 「新規APIエンドポイント追加が必要な時に使用。glow-schema確認からルーティング定義、Controller・ResultData・ResponseFactory実装、テストまでの全体フローを提供する。」
  - ❌ 「スキルを作成します。」

### Instructionsセクション

- [ ] **ステップ数は3-5（最大7）か**
  - 現在のステップ数: ___ステップ

- [ ] 各ステップの説明は1-2行に収まっているか

- [ ] 詳細な説明は参照ファイルに記載し、SKILL.mdには含めていないか

- [ ] 各ステップに適切な参照リンクがあるか

### 参照ドキュメントセクション

- [ ] guides/patterns/examples/ のカテゴリごとに分類されているか

- [ ] 各ファイルに簡潔な説明（1行）が付いているか

- [ ] ファイルパスが正しいか（相対パス）

### 関連スキルセクション（複数スキルの場合）

- [ ] 関連スキルセクションがあるか

- [ ] 典型的な使用フローが記載されているか

- [ ] 各関連スキルへのリンクが正しいか

## ファイル構成の検証

### ディレクトリ構造

- [ ] **必要なディレクトリが作成されているか**
  - guides/ (コンセプト・ツール解説)
  - patterns/ (実装パターン・方法論)
  - examples/ (具体的な実装例)

- [ ] **不要なディレクトリがないか**
  - ファイル数が5個以下の場合、ディレクトリ分割は不要

### ファイル命名

- [ ] **ファイル名はケバブケース（kebab-case）か**
  - ✅ 良い例: `new-table.md`, `column-changes.md`
  - ❌ 悪い例: `newTable.md`, `column_changes.md`

- [ ] **ディレクトリ名は複数形か**
  - ✅ 良い例: `guides/`, `patterns/`, `examples/`
  - ❌ 悪い例: `guide/`, `pattern/`, `example/`

### ファイルサイズ

- [ ] **各参照ファイルは200行程度を目安としているか**
  - 300行を超えるファイルがある場合は分割を検討

- [ ] **100行を超えるファイルに目次があるか**

### 相対パス参照

- [ ] **SKILL.mdからサブディレクトリへの参照が正しいか**
  - 形式: `**[ファイル名](examples/xxx.md)**`

- [ ] **サブディレクトリ内のファイル間の参照が正しいか**
  - 形式: `**[ファイル名](../guides/xxx.md)**`

## 内容の検証

### guides/ ファイル

- [ ] **コンセプトやツールの使い方が説明されているか**
  - 「何ができるか」「どう動くか」の理解
  - 説明的・教育的な内容

- [ ] **実際のコードより概念の説明が中心か**

- [ ] **「〜とは」「〜ガイド」「〜の使い方」といったタイトルか**

### patterns/ ファイル

- [ ] **実装パターンや方法論が記載されているか**
  - 「どのように実装するか」の手順と判断基準

- [ ] **ステップバイステップの手順があるか**

- [ ] **使い分けの判断基準が明確か**

- [ ] **「〜の方法」「〜のパターン」「〜の実装」といったタイトルか**

### examples/ ファイル

- [ ] **具体的な実装例が記載されているか**
  - 「実際のコード」を提示、そのまま真似できる

- [ ] **ファイルパスが明記されているか**
  - 例: `api/database/migrations/xxx.php`

- [ ] **コピー&ペースト可能なコードがあるか**

- [ ] **「〜の例」「〜のサンプル」「〜での実装」といったタイトルか**

### 実例の扱い

- [ ] **既存コードを積極的に活用しているか**
  - 架空の例より実際のコード

- [ ] **コード全体を掲載しているか**
  - 一部省略していないか

- [ ] **ファイルパスが明記されているか**

### チェックリストの設計

- [ ] **具体的で実行可能な項目か**

- [ ] **Yes/Noで判断できるか**

- [ ] **重要な項目に★マークや「重要」が明記されているか**

## 検証後のアクション

### 問題が見つかった場合

1. **SKILL.mdの修正**
   - 行数が多すぎる場合は内容を参照ファイルに移動
   - descriptionが不明確な場合は書き直す
   - ステップ数が多すぎる場合は統合

2. **ファイル構成の修正**
   - ファイル名が不適切な場合はリネーム
   - ファイルサイズが大きすぎる場合は分割
   - 相対パスが間違っている場合は修正

3. **内容の修正**
   - 不足している情報を追加
   - 重複している内容を削除
   - 分かりにくい説明を改善
