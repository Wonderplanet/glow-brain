# テスト結果レポートテンプレート

admin動作確認テスト完了後に作成するレポートのテンプレートと記入例を提供します。

## 基本テンプレート

```markdown
# Admin動作確認レポート

## テスト対象

[実装内容の簡潔な説明]

## テスト環境

- **Admin URL**: http://localhost:{port}/admin
- **実行日時**: [タイムスタンプ]
- **実行者**: [テスト実施者名]
- **ブラウザ**: Chrome（chrome-devtools MCP経由）

## テストケース

### 1. [テストケース名]

- **目的**: [テストの目的]
- **手順**:
  1. [実施した操作1]
  2. [実施した操作2]
  3. [実施した操作3]
- **結果**: ✅ 成功 / ❌ 失敗 / ⚠️ 警告あり
- **備考**: [特記事項があれば]
- **スクリーンショット**: [ファイルパス]

### 2. [テストケース名]

...（以下同様に追加）

## 検出した問題

### 問題 1: [問題のタイトル]

- **種類**: エラー / 警告 / 表示異常 / 操作失敗
- **重要度**: 高 / 中 / 低
- **内容**: [詳細な説明]
- **発生箇所**: [URL、画面名、操作手順]
- **再現手順**:
  1. [手順1]
  2. [手順2]
  3. [手順3]
- **エラーメッセージ**:
  ```
  [エラーメッセージの内容]
  ```
- **スクリーンショット**: [ファイルパス]
- **推奨される対処**: [修正提案]

### 問題 2: [問題のタイトル]

...（複数の問題がある場合）

**または**

問題は検出されませんでした。

## コンソールログ

### エラー
```

## 記入例2: バリデーションエラーの検出例

```markdown
# Admin動作確認レポート

## テスト対象

ユーザー登録フォームのバリデーション変更
- メールアドレスの一意性チェックを追加
- パスワードの最小長を8文字に変更

## テスト環境

- **Admin URL**: http://localhost:8081/admin
- **実行日時**: 2025-01-30 15:00:00
- **実行者**: admin-browser-tester agent
- **ブラウザ**: Chrome（chrome-devtools MCP経由）

## テストケース

### 1. 正常値でのユーザー作成

- **目的**: 正常値でユーザー作成が成功することを確認
- **手順**:
  1. ユーザー新規作成フォームを開く
  2. 以下のデータを入力:
     - Email: "newuser@example.com"
     - Password: "password123" (8文字以上)
     - Name: "Test User"
  3. 保存ボタンをクリック
- **結果**: ✅ 成功
- **備考**: ユーザーが正常に作成された
- **スクリーンショット**: `.claude/tmp/user_create_success.png`

### 2. パスワード最小長バリデーション確認

- **目的**: パスワードが8文字未満の場合、エラーメッセージが表示されることを確認
- **手順**:
  1. ユーザー新規作成フォームを開く
  2. Password に "pass123" (7文字) を入力
  3. 他のフィールドは正常値を入力
  4. 保存ボタンをクリック
- **結果**: ✅ 成功
- **備考**: 期待通りバリデーションエラーが表示された
- **スクリーンショット**: `.claude/tmp/password_validation_error.png`

### 3. メールアドレス一意性バリデーション確認

- **目的**: 既存のメールアドレスを入力した場合、エラーメッセージが表示されることを確認
- **手順**:
  1. ユーザー新規作成フォームを開く
  2. Email に既存ユーザーのメールアドレス "admin@wonderpla.net" を入力
  3. 他のフィールドは正常値を入力
  4. 保存ボタンをクリック
- **結果**: ❌ 失敗
- **備考**: バリデーションエラーが表示されず、ユーザーが作成されてしまった
- **スクリーンショット**: `.claude/tmp/email_unique_validation_failed.png`

## 検出した問題

### 問題 1: メールアドレス一意性バリデーションが機能していない

- **種類**: エラー
- **重要度**: 高
- **内容**: 既存のメールアドレスを入力してもバリデーションエラーが表示されず、データが重複して保存される
- **発生箇所**: ユーザー新規作成フォーム (`/admin/users/create`)
- **再現手順**:
  1. ユーザー新規作成フォームを開く
  2. Email に既存ユーザーのメールアドレスを入力（例: "admin@wonderpla.net"）
  3. Password: "password123"、Name: "Duplicate User" を入力
  4. 保存ボタンをクリック
- **期待される動作**: 「このメールアドレスは既に使用されています」等のエラーメッセージが表示され、保存が失敗する
- **実際の動作**: エラーメッセージが表示されず、ユーザーが作成される（データベースエラーが発生する可能性あり）
- **エラーメッセージ**: なし（エラーメッセージが表示されるべき）
- **スクリーンショット**: `.claude/tmp/email_unique_validation_failed.png`
- **推奨される対処**:
  - Resourceクラスのバリデーションルールを確認
  - `unique:users,email` ルールが正しく設定されているか確認
  - モデルの`$fillable`プロパティを確認

## コンソールログ

### エラー

なし

### 警告

なし

## ネットワークリクエスト

### 失敗したリクエスト

❌ **重大な問題あり** - メールアドレス一意性バリデーションの修正が必要です

## 推奨される次のステップ

1. Resourceクラスのバリデーションルールを確認・修正
2. 修正後、再度テストを実施
3. データベースに重複データが作成されていないか確認
```

## レポート作成時の注意事項

1. **具体性**: 抽象的な表現を避け、具体的な情報を記載する
2. **再現性**: 問題が発生した場合、他の人が再現できる手順を記載する
3. **証拠**: スクリーンショットやログを必ず添付する
4. **優先度**: 問題の重要度を明確にする（高/中/低）
5. **推奨事項**: 問題に対する解決策を提案する

## スクリーンショットの命名規則

スクリーンショットファイルは以下の命名規則に従う：

```
{動作内容}_{結果}.png
```

例:
- `navigation_menu.png` - ナビゲーションメニューの表示
- `product_create_success.png` - Product作成成功
- `password_validation_error.png` - パスワードバリデーションエラー
- `products_table_layout_issue.png` - テーブルレイアウトの問題
