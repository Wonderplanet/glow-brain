# 運営設計書フォーマット改善 (ops-design-sheet-improvement)

## 概要

運営設計書のフォーマットを標準化し、リリースキー単位のマスタデータ生成をAI指示1コマンドで完結させる仕組みを構築する。

## 背景・経緯

マスタデータ作成フローは以下のステップで進む：
1. 開発仕様書ができる
2. 実装開始
3. DBテーブル作成
4. 実装完了
5. マスタデータ入力の手順書作成
6. データ投入シート作成
7. 運営仕様書ができる
8. **運営設計**
9. **運営設計をインプットしてマスタデータAI生成**
10. マスタデータ投入
11. 実機確認、QA
12. 必要ならデータ調整
13. 同じリリースキーのデータ全てで対応
14. リリース

このうち「運営設計書」フェーズに課題がある。

## 解決したい問題

### 問題1: 同じリリースキーの設計書が散在している
- 同じリリースキー（データデプロイタイミングが同じデータ）の設計書がどこにあるか整理されていない
- 一覧できる場所がない
- AIがスキルを選定する際に「このシートはどのデータが記載されているか？」を探索するトークンが無駄

### 問題2: 参照シートのノイズが多い
- 参照用のシートや直接的に関係ないシートが多く、AIコンテキストのノイズになっている
- 不要なシートを参照するだけで実体として含めたくない

## 期待する成果

1. **AI指示→マスタデータ生成が1コマンドで完結**: リリースキーを指定するだけで、必要なスキルが自動選定・実行される
2. **運営設計書の標準フォーマット完成**: どのスプシも同じ構造で、AIが迷わず読めるフォーマットが定まる

## ネクストアクション候補

- 標準フォーマット定義の設計・ドキュメント化
- 各スプシへの基本設定シート追加（スクリプト化）
- リリースキー×スキルの対応マッピング設計
- DuckDBでクエリしやすいテーブル形式への移行

## 解決アイデア

### 運営仕様書の構造改善
- 同じリリースキーのデータを1シートに全て記載する
- 運営仕様書にまとまっていると良い
  - シート形式に従って、AIに指示を投げて、適切なスキルを選定して、生成させることができる

### 基本設定シートの標準化
- 各スプシに「基本設定シート」を用意する
  - 用途・リリースキー・スキル名など、一意に識別できる情報を特定位置に配置
  - スクリプトで機械的に読み取れる状態にしておく
- 通常のテーブルデータ形式にしておく → DuckDBでクエリしやすい

### ノイズ削減
- 参照用シートは実体を置かず、参照するだけ
- もしくは参照専用シートに同じプレフィックスをつける（例: `ref_`）
- AIコンテキスト作成時のノイズを排除
