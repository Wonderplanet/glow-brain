# MstDummyUserUnit 詳細説明

> CSVパス: `projects/glow-masterdata/MstDummyUserUnit.csv`

---

## 概要

`MstDummyUserUnit` は**ダミーユーザーが所持しているユニット（キャラクター）の定義テーブル**。

マッチング用ダミー対戦相手として使用される仮想ユーザー（`MstDummyUser`）が所持するユニットのリストと、各ユニットのレベル・ランク・グレードレベルを管理する。クライアントではフォロワー一覧やPvP対戦相手のデッキ情報として使用される。

CSVの行数は828件（2026年3月現在）。ダミーユーザー1人あたり平均10〜20ユニットを所持している計算になる。

### ゲームプレイへの影響

- **ランク帯別の強さ表現**: `level`・`rank`・`grade_level` の組み合わせで、ダミーユーザーがランク帯相応の強さになるよう設定する
- **フォロワー表示**: ホーム画面やフォロワー一覧で表示されるダミーユーザーの所持ユニット表示に使用
- **PvP対戦**: 実プレイヤーが少ない場合のマッチング相手のデッキ構成として参照される

### 関連テーブルとの構造図

```
MstDummyUser（ダミーユーザー本体）
  └─ id → MstDummyUserUnit.mst_dummy_user_id（1:N、所持ユニットリスト）
                └─ mst_unit_id → MstUnit.id（ユニット本体）

MstDummyUser
  └─ id → MstDummyUserArtwork.mst_dummy_user_id（1:N、所持原画リスト）
```

## 設定時のポイント

1. **ランク帯に応じたユニットの強さ設定が重要**。`level`・`rank`・`grade_level` の組み合わせで実際のプレイヤーの平均的な強さに合わせること。低ランク帯のダミーユーザーに高育成ユニットを持たせると初心者プレイヤーが勝てなくなる。

2. **`id` は1始まりの連番整数**。CSVへの行追加時は既存最大値+1を使用する。他のテーブルと違いPascalCaseの文字列IDではない点に注意。

3. **1ダミーユーザーには複数ユニットを持たせることができる**。ユニーク制約はなく、同一ユニットを複数行設定することも技術的には可能だが、通常は重複させない。

4. **`mst_unit_id` は `mst_units` テーブルに存在するIDのみ指定可能**。存在しないIDを設定するとサーバーエラーの原因となる。

5. **`grade_level` は1から設定するのが一般的**。グレードレベルが上がるとスペシャルアタックが解放されるため、ダミーユーザーの強さ設計時に考慮する。

6. **`mst_dummy_user_id` の大文字・小文字表記に注意**。実データでは `bronze_0_user1`（小文字）と `Platinum_1_user1`（先頭大文字）が混在しているため、`MstDummyUser.csv` の既存IDと完全一致させること。
