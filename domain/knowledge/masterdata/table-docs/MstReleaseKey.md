# MstReleaseKey 詳細説明

> CSVパス: `projects/glow-masterdata/MstReleaseKey.csv`（※ CSVファイルは存在しないか非公開の可能性あり）

---

## 概要

`MstReleaseKey` は**マスタデータのバージョン管理キーを定義するテーブル**。各マスタデータCSVの `release_key` カラムに設定する値の定義と、そのリリース開始日時を管理する。

`release_key` は各マスタデータテーブルで共通利用される概念で、「どのリリース以降でそのデータが有効になるか」を制御する仕組み。クライアントはサーバーから自身のバージョンに対応するデータのみを受け取る。

### ゲームへの影響

- `release_key` の値が大きいほど新しいリリースのデータ。
- クライアントのバージョンが対応する `release_key` 以上のデータのみが配信される。
- 旧クライアントと新クライアントが同時運用されている期間（強制アップデート前）のデータ分離に使用する。

### リリースキーの命名規則

```
YYYYMMXXX
```

- `YYYY`: 年（例: 2025）
- `MM`: 月（例: 09）
- `XXX`: 月内の連番（例: 010, 020, 025）

実際に使用されているリリースキーの例:
| release_key | 意味 |
|-------------|------|
| `202509010` | 2025年9月 第1回リリース |
| `202510010` | 2025年10月 第1回リリース |
| `202510020` | 2025年10月 第2回リリース |
| `202511010` | 2025年11月 第1回リリース |
| `202511020` | 2025年11月 第2回リリース |
| `202512010` | 2025年12月 第1回リリース |
| `202512015` | 2025年12月 第2回リリース |
| `202512020` | 2025年12月 第3回リリース |
| `202601010` | 2026年1月 第1回リリース |
| `202602015` | 2026年2月 第2回リリース |
| `202602020` | 2026年2月 第3回リリース |
| `202603010` | 2026年3月 第1回リリース |
| `202603020` | 2026年3月 第2回リリース |
| `202603025` | 2026年3月 第3回リリース |
| `999999999` | 開発・テスト専用。本番では配信されない |

---

## 全カラム一覧

| カラム名 | 型 | NULL許容 | デフォルト値 | 説明 |
|---------|----|---------|-----------|----|
| `release_key` | int | 不可 | - | 主キー。リリースキー識別値 |
| `start_at` | datetime | 不可 | `2000-01-01 00:00:00` | このリリースキーが有効になる開始日時 |
| `description` | varchar(255) | 可（NULL） | - | リリース内容の説明文（任意） |

---

## 他テーブルとの連携

`release_key` は全マスタデータテーブルで共通利用される横断的なカラム。主な利用テーブル:

| テーブル | 用途 |
|--------|------|
| `MstQuest` | クエストデータの配信バージョン制御 |
| `MstStage` | ステージデータの配信バージョン制御 |
| `MstUnit` | ユニットデータの配信バージョン制御 |
| `MstQuestBonusUnit` | ボーナス設定の配信バージョン制御 |
| `MstShopPass` | ショップパスデータの配信バージョン制御 |
| その他全マスタテーブル | 各データのバージョン管理 |

## 実データ例

### パターン1: 初回リリース（開発・テスト含む）

```csv
release_key,start_at,description
202509010,2025-09-22 00:00:00,2025年9月 リリース1
999999999,2000-01-01 00:00:00,開発・テスト専用
```

- `999999999` はテスト用で本番クライアントには配信されない

### パターン2: 月内複数リリース

```csv
release_key,start_at,description
202510010,2025-10-06 00:00:00,2025年10月 リリース1
202510020,2025-10-27 00:00:00,2025年10月 リリース2
```

- 同月内に複数のリリースがある場合は連番（010, 020, ...）で区別する

## 設定時のポイント

1. **新しいリリースを追加する際は、まず `MstReleaseKey` にレコードを追加してから、各テーブルで該当 `release_key` を使用する**。`MstReleaseKey` に存在しない値を参照している場合、整合性エラーになる。
2. **`start_at` は実際のアプリリリース日時に合わせて設定する**。クライアントはこの日時を基にデータを有効化する。
3. **`release_key = 999999999` は開発・テスト専用**。本番環境には配信されないため、QA・動作確認用データに使用する。
4. **リリースキーは `YYYYMMXXX` 形式で命名する**。月内の連番は010, 020, 025のように10または5刻みで管理することが多い。
5. **同一リリースに含めるデータは同じ `release_key` を設定する**。複数テーブルにまたがる機能追加時は、全テーブルで同じキーを使用する。
6. **リリースキーは後から変更しないのが原則**。データ公開後に `release_key` を変更すると、旧クライアントへのデータ配信に影響が出る。
